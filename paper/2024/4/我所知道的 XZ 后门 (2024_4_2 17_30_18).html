我所知道的 XZ 后门

1小时之前 2024年04月02日
威胁情报

目录

译者：知道创宇404实验室翻译组
原文链接：Everything I Know About the XZ Backdoor

2021

JiaT75（Jia Tan）创建了自己的GitHub账号，并提交了第一个与XZ无关但极具可疑性的PR。在libarchive中，他们提出了一个PR：使用bsdtar解压添加错误文本到警告中。然而，这个PR实际上涉及更多内容。他们替换了safe_fprint函数为一个不安全的变体，这可能导致另一个漏洞的出现。令人担忧的是，这段代码在没有任何讨论的情况下被合并，且至今仍在使用（已修复）。

2022

2022年4月，Jia Tan通过邮件列表提交了一个与事件无关的补丁。随后，一个名为Jigar Kumar的新人开始施压，要求合并该补丁。

不久之后，Jigar Kumar开始向Lasse Collin施加压力，要求将另一名维护者加入XZ项目。在这一过程中，我们对开源项目中的心理健康有了一些了解。

在对Lasse Collin施加压力的邮件发出三天后，JiaT75在XZ项目中提交了他们的第一个补丁：Tests: Created tests for hardware functions.。此后，他们成为了XZ项目的常规贡献者，目前排名第二。

Jigar Kumar再未出现过，而Dennis Ens参与了压力活动，其电子邮件地址格式与Kumar相似，但两者在XZ讨论之外从未出现，且也没有发现任何相关联的账户。

我真诚希望这能引发整个行业对常见做法的反思：即让整个产品的重任都压在一个过度劳累的人身上，而没有得到任何财务或运营上的支持，他们正悄然陷入心理健康危机。我希望所有依赖开源的人都能阅读此消息：邮件链接。

2023

2023年1月7日，JiaT75合并了他们的第一个提交，这表明他们开始获得信任。3月份，Google oss-fuzz项目的主要联系邮箱被更新，由Lasse Collin的邮箱变更为Jia的邮箱。

测试基础设施将用于漏洞利用的提交。尽管Lasse Collin被认为是该文的作者，但Jia Tan却犯了这个错误，最初是由Hans Jansen在6月份编写的。

  • 提交：liblzma: Add ifunc implementation to crc64_fast.c
  • PR：Replaced crc64_fast constructor with ifunc by hansjans162

Hans Jansen的账户似乎是专门用来创建拉取请求的，活动很少。稍后，他们将推动将受损版本包含在Debian中。

7月，由于上述更改引入的问题，oss-fuzz开了一个禁用ifunc的PR来进行模糊测试构建，而这似乎是为了掩盖即将引入的恶意更改。此外，JiaT75还在clang的问题中提到了一个警告，虽然不正确，但引起了对ifunc的关注。

2024

打开 Google oss-fuzz 项目的拉取请求，将项目的 URL 从tukaani.org/xz/更改为xz.tukaani.org/xz-utils/。tukaani.org托管在芬兰的 5.44.245.25，并由 Zoner 公司进行托管。与此同时，xz 子域指向 GitHub 页面。这增加了 Jia 对项目的控制。

包含执行此后门所需的最后步骤的提交被添加到了仓库中：

  • Tests: Add a few test files
  • Tests: Update two test files
发现

一封邮件被发送到oss-security邮件列表： backdoor in upstream xz/liblzma leading to ssh server compromise, 宣布了这一发现，并尽最大努力解释了攻击链。

我正在进行微基准测试，需要让系统静止以减少噪音。注意到sshd进程消耗大量CPU，尽管由于用户名错误等原因立即失败。对sshd进行了性能分析，发现在liblzma中消耗了大量CPU时间，但无法通过perf归因于特定符号。这引起了怀疑。回忆起几周前软件包更新后，自动测试PostgreSQL时看到过奇怪的valgrind警告。这确实需要很多巧合。

突然推进

Hans提出了一个请求，要求将受漏洞影响的版本包含在Debian中：

  • #1067708 - XZ-utils：可用的新上游版本

此请求是在Hans的Debian GitLab账户创建的同一周开启的。在请求之后，该账户在一些低流量存储库中创建了一些类似的“更新”请求，以建立可信度。

另外，一些其他可疑的匿名账户，如misoeater91和krygorin4545，也在推动加入，这些账户之前几乎没有活动记录。krygorin4545的PGP密钥是在加入讨论两天前创建的。

有人发现了这个bug，额外的Valgrind输出导致了一些测试失败。新版本看起来会解决这个问题，希望能尽快得到新版本，这样工作可以继续进行。另外，另一位用户上周也注意到了这个问题，差点就要提交一个Valgrind bug报告。很高兴看到问题被解决了，感谢Hans！

所提到的Valgrind bug是由这个恶意注入引入的，正如在发送给OSS-Security的电子邮件中所述：

由于堆栈布局与后门期望的不同，注入的代码（更多信息见下文）在某些配置中导致了 valgrind 错误和崩溃。这些问题已在 5.6.1 中尝试解决：

一名1Password员工向一个go库提出了拉取请求，请求升级到受漏洞影响的版本，然而，这个请求的提出时间不太幸运。1Password通过电子邮件联系我，并引用了这个评论。

一位Fedora贡献者表示，Jia正在推动将受漏洞影响的版本包含在Fedora中，因为它包含了一些“很棒的新功能”。

几小时后，GitHub暂停了JiaT75的账户并禁止了仓库，这意味着人们无法再在不借助镜像的情况下审核对其所做的更改。此举备受称赞。他们还暂停了Lasse Collin的账户，这一做法是不可接受的。

Lasse已经开始恢复Jia引入的更改，包括一个禁用沙盒的隐藏句点。他们还发布了一个常见问题解答，开始解释这种情况：XZ Utils backdoor

OSINT

收到了关于Jia身份的发现的多个联系。其中一些已包含在时间轴中，但还有其他的是新的内容。

IRC

我收到一封电子邮件，澄清了一些问题并提供了新的见解。

邮件指出，"Jia Tan"曾在Libera.chat上的#tukaani IRC频道中出现过。通过/whois命令可以查看到他们的连接IP以及3月29日的活动。

[libera] -!- jiatan [~jiatan@185.128.24.163]
[libera] -!-  was      : Jia Tan
[libera] -!-  hostname : 185.128.24.163
[libera] -!-  account  : jiatan
[libera] -!-  server   : tungsten.libera.chat [Fri Mar 29 14:47:40 2024]
[libera] -!- End of WHOWAS

对该IP进行了nmap扫描，发现了许多开放端口，这可能表明它是一个代理、托管提供商或类似的服务。该IP来自新加坡。

进一步研究显示，该IP属于Witopia VPN，因此并不完全反映地理位置。然而，考虑到时区，与新加坡的接近性是合理的。

LinkedIn上的几点说明

我收到了几封邮件，提醒我关注LinkedIn上的一个名叫Jia Tan的人。在简介中夸耀自己有大规模漏洞管理的经验，声称居住在加利福尼亚。但JiaT75在GitHub上的提交时间被设置为UTC-0800，并不符合加利福尼亚的时区UTC-0800。大多数提交在UTC 12-17之间，这对于加利福尼亚来说实在太早了。还没有足够的证据表明该LinkedIn账户是我们的目标。我认为更可能是身份盗窃，但我愿意接受更多证据。

Git 日志发现

我收到了一封来自Minhu Wang的电子邮件，他调查了 Git 日志，发现Jia的用户名不同：

$ git shortlog --summary --numbered --email | grep grep jiat0218@gmail.com
273 Jia Tan <jiat0218@gmail.com>
2 jiat75 <jiat0218@gmail.com>
1 Jia Cheong Tan <jiat0218@gmail.com>

发现了有关“Cheong”的有趣信息。现在另一个消息源指出，“Cheong”不是普通话，而是粤语。据推测，“Cheong”可能是“张”姓的变体，在香港作为官方姓氏罗马拼音相当常见。第三个消息来源提醒我，“Jia”是普通话，因为粤语很少使用J，尤其不使用Ji。姓“Tan”在普通话中是可能的，但最常见的是闽南话中对“陈”这个字的发音。我们的黑客很可能只是把听起来合理的中文名字拼在一起。

另外，对提交时间的独立分析得出结论，犯罪者在UTC+02/03时区的“工作时间”内进行了工作。特别注意的是，他们在农历新年期间工作，以及一些重要的东欧假日（包括圣诞节和新年），并没有工作。
